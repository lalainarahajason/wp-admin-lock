# F1 — Custom Admin URL
**Priorité : HAUTE**

## Description
Remplacer l'URL d'accès `/wp-admin` et `/wp-login.php` par une URL personnalisée pour empêcher les bots d'atteindre la page de login. La redirection sur l'ancienne URL renvoie une 404 (pas une 403) pour ne pas confirmer l'existence de l'admin. Les appels internes WordPress (Gutenberg, AJAX, WooCommerce) continuent de fonctionner sans modification. Obfusquer le point d'entrée de l'administration pour stopper 99% des attaques automatisées de brute-force.

## Classe : `LBS_AdminUrl`
Fichier : `plugin/includes/Features/AdminUrl.php`

## Comportement attendu

- Accès à `/wp-admin` ou `/wp-login.php` depuis l'extérieur → HTTP 404 (pas 403)
- Accès au slug custom → page de connexion WordPress normalement
- `admin-ajax.php` jamais bloqué (AJAX WordPress)
- Gutenberg, WooCommerce, tous les plugins admin fonctionnent normalement
- Le slug est stocké en base (`wp_options`), modifiable sans accès FTP

## Config JSON

```json
"admin_url": {
  "enabled": true,
  "slug": "mon-espace-admin"
}
```

## Mécanismes de récupération d'urgence (ordre de priorité)

Trois niveaux de bypass, du plus simple au plus lourd. Ils s'activent indépendamment et se cumulent si besoin.

### Niveau 1 — Token de récupération (sans accès serveur)
Un token secret à usage unique est généré à l'activation du plugin et stocké dans `wp_options` (`lbs_recovery_token`). Il permet d'accéder à `/wp-login.php?lbs_recover={token}` sans connaître le slug custom. **Le token est invalidé et régénéré après chaque utilisation.**

```php
// Génération du token (à l'activation)
$token = bin2hex(random_bytes(32));
update_option('lbs_recovery_token', $token);

// Vérification dans le hook init
$token_from_url = sanitize_text_field( wp_unslash( $_GET['lbs_recover'] ?? '' ) );
$stored_token   = get_option('lbs_recovery_token', '');

if ( $stored_token && hash_equals($stored_token, $token_from_url) ) {
    // Invalider immédiatement le token
    update_option('lbs_recovery_token', bin2hex(random_bytes(32)));
    // Laisser passer vers wp-login.php
    return;
}
```

> ⚠️ **Le token doit être affiché une seule fois à l'activation** dans un admin notice avec instruction de le copier. Il est également accessible depuis la page de settings du plugin pour les admins connectés.

### Niveau 2 — Constante wp-config (accès FTP/SFTP requis)
Si `define('LBS_DISABLE_CUSTOM_URL', true);` est présent dans `wp-config.php`, le slug custom est intégralement ignoré et `/wp-login.php` répond normalement. À utiliser en cas de perte du token.

### Niveau 3 — Désactivation du plugin (accès FTP/SFTP requis)
En dernier recours, renommer le dossier du plugin (`lebo-secu` → `lebo-secu-disabled`) désactive tous les hooks. Procédure documentée dans le README du plugin.

## Hooks WordPress à utiliser

- `init` (priorité 1) — intercepter les requêtes entrantes, évaluer les bypasses, bloquer ou laisser passer
- `site_url` / `admin_url` / `network_site_url` / `wp_redirect` — réécrire les URLs générées par WordPress

## Logique d'interception (hook `init`, priorité 1)

L'interception suit un ordre strict. Dès qu'une condition autorise le passage, on `return` immédiatement sans évaluer les suivantes.

```
1. DOING_CRON || DOING_AJAX || defined('DOING_AUTOSAVE')  → laisser passer
2. defined('LBS_DISABLE_CUSTOM_URL')                      → laisser passer
3. Token de récupération valide dans $_GET['lbs_recover']  → laisser passer + invalider token
4. Requête vers admin-ajax.php ou async-upload.php         → laisser passer
5. $_GET['action'] présent                                 → laisser passer
6. Requête vers slug custom                                → laisser passer (servir wp-login.php)
7. Requête vers /wp-login.php ou /wp-admin                 → wp_die() avec status 404, exit
   (ne pas charger le thème — utiliser wp_die() avant wp() )
```

> **Règle critique sur le 404** : ne pas appeler `include(get_404_template())` car cela charge le thème complet et exécute `wp()`. Utiliser `wp_die('', '', ['response' => 404])` suivi d'un `exit` pour un 404 léger, sans rendu thème.

## Contraintes

- Valider que le slug n'est pas un slug réservé WordPress (`wp-admin`, `wp-content`, `wp-includes`, `wp-json`, `wp-login.php`, `wp-cron.php`, etc.) — liste exhaustive dans `LBS_Helpers::get_reserved_slugs()`
- Valider que le slug n'entre pas en conflit avec un slug de page WordPress existant (`get_page_by_path($slug)`)
- Ne jamais bloquer les requêtes AJAX (`DOING_AJAX` ou `$_GET['action']` présent)
- Ne pas bloquer les cron WordPress (`DOING_CRON`)
- Ne jamais bloquer `admin-ajax.php` ou `async-upload.php`
- Filtrer l'URL via les hooks `site_url`, `network_site_url`, `admin_url` et `wp_redirect`
- Le slug doit être modifiable depuis l'interface admin sans accès FTP ; après modification, le nouveau slug est actif immédiatement (pas de flush des permaliens requis)

## Critères d'acceptation

- [ ] L'URL personnalisée donne accès au tableau de bord WordPress normalement
- [ ] Accès à `/wp-admin` et `/wp-login.php` depuis l'extérieur retourne une 404 **légère** (sans rendu thème)
- [ ] Les requêtes AJAX (`admin-ajax.php`) ne sont pas bloquées
- [ ] Gutenberg et les plugins admin-dépendants fonctionnent normalement
- [ ] L'URL est configurable depuis l'interface du plugin
- [ ] Validation anti-conflit : refus des slugs réservés WordPress
- [ ] Validation anti-conflit : refus si un slug de page existant correspond
- [ ] L'URL custom est persistante après une mise à jour des permaliens
- [ ] **Token de récupération** affiché à l'activation dans un admin notice
- [ ] **Token de récupération** accessible depuis la page de settings (admins connectés uniquement)
- [ ] **Token invalidé et régénéré** après chaque utilisation
- [ ] `define('LBS_DISABLE_CUSTOM_URL', true)` désactive intégralement la feature
- [ ] Une tentative d'accès à `/wp-login.php` ne charge pas le moteur de rendu du thème

## Risques et points de vigilance

| Risque | Mitigation |
|---|---|
| Admin oublie le slug custom | Token de récupération à usage unique |
| Token de récupération intercepté | Usage unique + régénération immédiate |
| Conflit avec une page existante | Validation au save + message d'erreur explicite |
| Hébergement avec cache full-page (Varnish, LiteSpeed) | Documenter l'exclusion du slug custom des règles de cache |
| Permalink flush efface la règle de réécriture | Utiliser `init` pour l'interception, pas `rewrite_rules` |
