
### F10 — Journal d'activité (Audit Log)
**Priorité : MOYENNE**

## Objectif
Logger les événements de sécurité importants dans une table dédiée (`wp_lebosecu_logs`). Interface de consultation avec filtres, pagination et export CSV. Rétention automatique configurable.

**Critères d'acceptation**
[ ] Schéma Extensible : Table wp_lebosecu_logs créée à l'activation avec colonnes : event_code, severity, actor_id, actor_ip, metadata (JSON).

[ ] Capture Automatique : Le logger récupère dynamiquement l'IP (gestion proxy/Cloudflare), le User-Agent et l'URL de la requête sans intervention manuelle.

[ ] Audit de Config : Chaque modification via l'interface du plugin génère un log INFO avec metadata contenant les anciennes et nouvelles valeurs.

[ ] Alertes Critiques : Enregistrement immédiat de l'usage du token de secours (F1) et des blocages de sécurité (F4, F6, F7) avec sévérité CRITICAL.

[ ] UI Actionnable : Table React avec filtres par sévérité/IP et bouton "Quick Ban" pour les IPs malveillantes.

[ ] Housekeeping Intelligent : Cron job de rétention (30j par défaut) ET protection contre la saturation (limite de taille de table ou agrégation des logs identiques).

[ ] Conformité RGPD : Option pour anonymiser (hacher) les IPs après une période définie.

[ ] Export Exportable : Export CSV respectant les filtres appliqués à l'écran.

## 1. Modèle de Données (Database)
La table `wp_lebosecu_logs` doit être optimisée pour l'écriture rapide (indexation stratégique) et la lecture filtrée.

| Colonne | Type | Index | Description |
| :--- | :--- | :---: | :--- |
| `id` | BIGINT(20) | PK | Auto-incrément. |
| `created_at` | DATETIME | Yes | Timestamp précis (UTC). |
| `event_code` | VARCHAR(50) | Yes | Identifiant unique de l'action (ex: `AUTH_FAILED`). |
| `severity` | ENUM | Yes | `INFO`, `NOTICE`, `WARNING`, `CRITICAL`. |
| `actor_id` | BIGINT(20) | Yes | ID de l'user WP (0 si anonyme/bot). |
| `actor_ip` | VARCHAR(45) | Yes | Support IPv4 et IPv6. |
| `user_agent` | TEXT | No | Identité du navigateur/bot. |
| `request_url` | TEXT | No | L'URL exacte visée (utile pour F1 et F4). |
| `metadata` | LONGTEXT | No | JSON stockant les détails variables (ex: ancien vs nouveau slug). |

## 2. Dictionnaire des Événements (Ontologie)
Tu dois utiliser ces codes standards :

### Sécurité (WARNING / CRITICAL)
* `AUTH_LOGIN_SUCCESS` / `AUTH_LOGIN_FAILED` : Tentatives de connexion.
* `AUTH_RECOVERY_TOKEN_USED` : **CRITICAL**. Usage du bypass F1.
* `AUTH_LOCKOUT` : IP bannie temporairement par F6.
* `SECURITY_ENUMERATION_BLOCKED` : Tentative `?author=n` stoppée (F7).
* `SECURITY_REST_DENIED` : Requête API REST rejetée (F4).

### Configuration (INFO / NOTICE)
* `CONFIG_UPDATED` : Changement d'un toggle ou d'une option.
* `CONFIG_IMPORTED` / `CONFIG_EXPORTED` : Mouvements via F5.
* `SYSTEM_HTACCESS_MODIFIED` : Écriture dans le fichier par F3.

## 3. Architecture Logicielle (PHP)
Tu ne dois pas instancier la DB partout. Crée une classe statique ou un Singleton pour le logging.

```php
/**
 * Exemple de signature pour l'IA
 * LBS_AuditLog::log( string $event_code, string $severity, array $metadata = [] );
 */
```
**Règle d'or :** La méthode `log()` doit récupérer automatiquement `actor_id`, `actor_ip`, `user_agent` et `request_url` via les helpers WP pour éviter de répéter le code.

## 4. Interface Utilisateur (UI/UX)
La page `AuditLogPage.tsx` doit permettre de transformer une information en action.

### Fonctions de la Table :
* **Groupement par IP :** Un clic sur une IP doit filtrer tous les logs de cette IP.
* **Action "Quick Ban" :** À côté d'une IP suspecte, bouton "Bannir" (ajoute l'IP à la blacklist de F6).
* **Visualisation JSON :** Les `metadata` doivent être rendus de manière lisible (pas un dump brut).
* **Indicateurs visuels :** Pastille de couleur selon la sévérité (Rouge pour Critical, Jaune pour Warning).

### Maintenance (Le "Housekeeping") :
* **Rétention tournante :** Un cron job (`wp_schedule_event`) supprime les logs plus vieux que X jours (défini en config, défaut 30).
* **Alerte de taille :** Si la table dépasse 100 000 lignes, afficher une notice admin pour suggérer une purge.

## 5. Ce que tu as tendance à oublier (The Mirror Part)
* **Performance :** Sur un site à fort trafic sous attaque, insérer un log à chaque hit peut tuer le serveur. **Solution :** Si le plugin détecte une attaque brute-force massive (F6 actif), il peut passer en mode "Aggregated Logging" (un seul log pour "1000 attaques bloquées" au lieu de 1000 lignes).
* **Anonymisation :** Pour la conformité RGPD, ajoute une option "Hash IPs" qui transforme `192.168.1.1` en un hash non réversible après 7 jours.
* **Fausse Sécurité :** Si un hacker prend le contrôle de la DB, il peut effacer ses propres logs. **Conseil Advisor :** Ajoute une fonction d'export distant (Webhook) dans une v1.1 pour envoyer les logs `CRITICAL` vers un service externe en temps réel.

# Mise à jour Spécification : Gemini CLI Context for Lebo Secu (F10 - Audit Log)

## 1. Objectif de la mise à jour
Garantir que l'IA génère un journal d'audit (F10) **cohérent, extensible et performant**, en standardisant la capture des données et en introduisant une structure de détails enrichie (JSON metadata).

## 2. Directives de Développement (Nouvelles Règles Strictes)

### A. Implémentation du Logging (F10 Injector)
* **Systématisation :** Toute action de sécurité critique (F1, F4, F6, F7, F9) et toute modification de configuration (F5) **doit** déclencher un appel au Logger Engine.
* **Le "Quoi" (event_code) :** Tu ne dois **jamais** utiliser de chaînes de caractères brutes ("magic strings") pour les codes d'événements. Elle doit exclusivement utiliser les constantes définies dans `includes/LBS_EventCodes.php`.
* **La Sévérité (severity) :** L'IA doit attribuer le niveau de sévérité correct (`INFO`, `NOTICE`, `WARNING`, `CRITICAL`) selon la nature de l'événement (ex: `AUTH_RECOVERY_TOKEN_USED` est `CRITICAL`, `CONFIG_UPDATED` est `INFO`).

### B. Standardisation des Données (Dictionnaire des Événements)
L'IA doit se référer à ce dictionnaire pour garantir la cohérence des filtres dans l'UI :

| Code Événement (Constante) | Sévérité par défaut | Description |
| :--- | :---: | :--- |
| `LBS_EventCodes::AUTH_LOGIN_SUCCESS` | `INFO` | Connexion réussie. |
| `LBS_EventCodes::AUTH_LOGIN_FAILED` | `WARNING` | Échec de connexion. |
| `LBS_EventCodes::AUTH_RECOVERY_TOKEN_USED` | `CRITICAL` | Usage du token de secours (F1). |
| `LBS_EventCodes::AUTH_LOCKOUT` | `CRITICAL` | IP bannie temporairement (F6). |
| `LBS_EventCodes::SECURITY_ENUMERATION_BLOCKED`| `WARNING` | Tentative d'énumération stoppée (F7).|
| `LBS_EventCodes::CONFIG_UPDATED` | `INFO` | Changement de réglage (F5). |
| `LBS_EventCodes::SYSTEM_HTACCESS_WRITTEN` | `INFO` | Écriture .htaccess réussie (F3). |

### C. Structure des Détails (JSON Metadata Enrichment)
L'IA ne doit pas se contenter d'un JSON vide ou minimal. Pour chaque événement, elle doit enrichir le champ `metadata` avec le contexte pertinent.

* **Exemple pour `AUTH_LOGIN_SUCCESS` :**
```json
{
  "username": "admin",
  "auth_method": "wp-login.php", // ou "xmlrpc.php"
  "auth_type": "standard", // ou "application_password"
  "remember_me": true
}
```
* **Exemple pour `CONFIG_UPDATED` :**
```json
{
  "source": "settings_page", // ou "rest_api"
  "changes": {
    "lbs_enable_login_protection": { "old": 0, "new": 1 }
  }
}
```
