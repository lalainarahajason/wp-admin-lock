# F6 — Protection de la page de login
**Priorité : HAUTE**

## Objectif
Limiter le nombre de tentatives de connexion par IP. Blocage temporaire après N échecs consécutifs (seuil et durée configurables). Option HTTP Basic Auth devant `/wp-login.php`. Notification email optionnelle sur blocage.

## Spécifications

- **Seuil par défaut** : 5 tentatives
- **Durée de blocage par défaut** : 15 minutes (900s)
- **Stockage** : Deux transients WP par IP, séparés par responsabilité (voir ci-dessous)
- **Nettoyage** : Expiration automatique via TTL des transients — aucune maintenance de table

## Classe : `LBS_LoginProtection`
Fichier : `plugin/includes/Features/LoginProtection.php`

## Stratégie de stockage (transients, pas wp_options)

> **Pourquoi pas `wp_options` avec un JSON global ?**
> Un site sous attaque brute force peut recevoir des dizaines de requêtes concurrentes par seconde. Stocker toutes les IPs dans un seul `wp_options` JSON implique un read-modify-write sur la même ligne à chaque tentative — race condition garantie. Deux requêtes simultanées lisent `count: 3`, incrémentent toutes les deux à `4`, et une tentative est "perdue". Le blocage n'intervient jamais, ou trop tard.

**Deux transients par IP**, avec TTL distincts :

| Transient | Clé | TTL | Contenu |
|---|---|---|---|
| Compteur de tentatives | `lbs_attempts_{hash_ip}` | `lockout_duration` | `int` — nombre d'échecs |
| Verrou actif | `lbs_lock_{hash_ip}` | `lockout_duration` | `int` timestamp du blocage |

```php
// Hash de l'IP pour éviter les caractères problématiques dans les clés de transient
$key = 'lbs_attempts_' . substr(md5($ip), 0, 16);
```

**Pourquoi deux transients séparés ?**
- Le compteur a un TTL glissant (reset à chaque tentative) ; le verrou a un TTL fixe depuis le moment du blocage.
- Séparer les deux permet de `delete_transient('lbs_lock_...')` pour un déblocage manuel sans toucher au compteur.
- WordPress garantit l'atomicité sur un transient individuel via l'autoload et l'index MySQL — pas sur un JSON partagé.

**Incrément atomique avec `get` + `set` protégé :**

```php
private function increment_attempts( string $ip ): int {
    $key   = $this->attempts_key($ip);
    $count = (int) get_transient($key) + 1;
    // TTL glissant : chaque tentative repousse l'expiration
    set_transient($key, $count, $this->config['lockout_duration']);
    return $count;
}
```

> **Note** : Sur des configs WordPress avec object cache externe (Redis, Memcached), les transients sont stockés dans le cache, pas en BDD — ce qui rend l'incrément non-atomique. Si le projet vise des sites avec object cache, envisager `wp_cache_incr()` à la place pour PHP 8.1+ avec Redis. En v1, documenter cette limitation.

## Mécanisme

1. Hook `authenticate` (priorité 1) → vérifier si l'IP est verrouillée **avant** toute authentification
2. Hook `wp_login_failed` → incrémenter le compteur ; poser le verrou si seuil atteint
3. Déblocage automatique à l'expiration du transient `lbs_lock_{hash_ip}`
4. Déblocage manuel depuis l'interface admin (delete_transient sur le verrou)

```php
add_filter('authenticate', [$this, 'check_ip_lockout'], 1, 3);
add_action('wp_login_failed', [$this, 'record_failed_attempt']);
```

## Implémentation détaillée

```php
public function check_ip_lockout( $user, string $username, string $password ) {
    $ip = $this->get_client_ip();

    // Ne jamais bloquer les IPs en whitelist
    if ( $this->is_whitelisted($ip) ) {
        return $user;
    }

    $lock_key = $this->lock_key($ip);

    if ( false !== get_transient($lock_key) ) {
        $locked_at      = (int) get_transient($lock_key);
        $retry_after    = $locked_at + $this->config['lockout_duration'] - time();
        header('Retry-After: ' . max(0, $retry_after));
        // Retourner une WP_Error arrête l'authentification proprement
        return new WP_Error(
            'too_many_attempts',
            sprintf(
                __('Trop de tentatives. Réessayez dans %d minutes.', 'lebo-secu'),
                ceil($retry_after / 60)
            )
        );
    }

    return $user;
}

public function record_failed_attempt( string $username ): void {
    $ip    = $this->get_client_ip();

    if ( $this->is_whitelisted($ip) ) {
        return;
    }

    $count = $this->increment_attempts($ip);

    if ( $count >= $this->config['max_attempts'] ) {
        $lock_key = $this->lock_key($ip);
        // Poser le verrou avec timestamp pour calculer Retry-After
        set_transient($lock_key, time(), $this->config['lockout_duration']);

        if ( $this->config['email_notify'] ) {
            $this->send_lockout_notification($ip, $username);
        }
    }
}
```

## Récupération de l'IP cliente

```php
private function get_client_ip(): string {
    // Ordre : header de proxy de confiance → IP directe
    // Ne pas faire confiance à X-Forwarded-For sans vérification du proxy
    $ip = $_SERVER['REMOTE_ADDR'] ?? '0.0.0.0';

    // Si derrière un proxy connu (Cloudflare, load balancer),
    // lire CF-Connecting-IP ou X-Real-IP uniquement si configuré explicitement
    // Ne JAMAIS lire X-Forwarded-For brut — vecteur de spoofing trivial

    return sanitize_text_field( wp_unslash( $ip ) );
}
```

> ⚠️ **Piège classique** : lire `HTTP_X_FORWARDED_FOR` sans whitelist du proxy permettrait à n'importe qui de spoofer son IP avec un header forgé et de contourner le rate limiting. En v1, se limiter à `REMOTE_ADDR`. Documenter la config proxy dans le README pour les sites derrière Cloudflare.

## Déblocage manuel (interface admin)

```php
// Liste des IPs actuellement bloquées — pour affichage admin
public function get_locked_ips(): array {
    global $wpdb;
    // Chercher tous les transients de verrou dans wp_options
    $rows = $wpdb->get_results(
        $wpdb->prepare(
            "SELECT option_name, option_value FROM {$wpdb->options}
             WHERE option_name LIKE %s",
            $wpdb->esc_like('_transient_lbs_lock_') . '%'
        )
    );
    // Reconstruire IP → timestamp depuis les clés hashées
    // (nécessite de stocker l'IP en clair dans la valeur du transient)
    return $rows;
}

// Déblocage d'une IP depuis l'interface admin
public function unlock_ip( string $ip ): void {
    delete_transient( $this->lock_key($ip) );
    delete_transient( $this->attempts_key($ip) );
}
```

> **Note sur la liste des IPs bloquées** : les clés de transient sont hashées, ce qui empêche de retrouver l'IP depuis la clé. Stocker l'IP en clair dans la **valeur** du transient `lbs_lock_` (ex: `['ip' => '1.2.3.4', 'locked_at' => timestamp]`) pour permettre l'affichage admin.

```json
"login_protection": {
  "enabled": true,

  "email_notify": false,

```

## Option HTTP Basic Auth

Si activée, déléguer l'écriture dans `.htaccess` à F3 (HtaccessManager) :

```apache
<Files "wp-login.php">
  AuthType Basic
  AuthName "Restricted"
  AuthUserFile /chemin/absolu/.htpasswd
  Require valid-user
</Files>
```

> Le chemin vers `.htpasswd` doit être **hors de la racine web** (`/etc/apache2/.htpasswd` ou équivalent). Ne jamais le placer dans `wp-content/`.

## Contraintes

- Ne jamais bloquer les cron WordPress (`DOING_CRON`)
- Ne jamais bloquer les IPs en whitelist, quelle que soit la phase
- L'email de notification utilise `wp_mail()` uniquement
- La whitelist IP doit être configurée au premier déploiement (IP du développeur)
- Sur des sites derrière Cloudflare : documenter la config `REMOTE_ADDR` vs `CF-Connecting-IP`

## Critères d'acceptation

- [ ] Comptage des tentatives par IP via transients (un transient par IP)
- [ ] Aucune race condition : deux requêtes simultanées ne peuvent pas toutes les deux lire un count périmé
- [ ] Blocage après seuil configurable (défaut : 5 tentatives / 15 min)
- [ ] Header `Retry-After` envoyé sur les réponses bloquées
- [ ] Réponse HTTP 429 (Too Many Requests) sur blocage — pas 403
- [ ] Déblocage automatique à l'expiration du transient
- [ ] Déblocage manuel depuis l'interface admin (par IP)
- [ ] Liste des IPs actuellement bloquées visible dans l'interface admin
- [ ] Option HTTP Basic Auth activable (délégation à F3)
- [ ] Notification email sur blocage (optionnel, via `wp_mail()`)
- [ ] Whitelist IP opérationnelle — les IPs whitelistées ne sont jamais comptabilisées
- [ ] La whitelist protège contre l'auto-lockout du développeur

## Risques et points de vigilance

| Risque | Mitigation |
|---|---|
| Race condition sur l'incrément | Transients individuels par IP (pas de JSON global) |
| Spoofing IP via X-Forwarded-For | Lire uniquement `REMOTE_ADDR` en v1 |
| Sites derrière Cloudflare | Documenter + option `trusted_proxy_header` en v1.1 |
| Object cache Redis (non-atomique) | Documenter la limitation ; `wp_cache_incr()` en v1.1 |
| Développeur auto-locké | Whitelist IP obligatoire au premier déploiement |
| .htpasswd accessible en web | Chemin hors racine web obligatoire |
