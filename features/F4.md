# F4 — Protection de l'API REST WordPress
**Priorité : HAUTE**

## Objectif
Verrouiller l'accès aux données du site via JSON, sauf pour les besoins légitimes du site.
Bloquer les requêtes non authentifiées provenant de l'extérieur vers l'API REST. Le serveur lui-même (nonce valide, utilisateur authentifié) conserve un accès complet. Gutenberg et les plugins utilisant l'API en interne ne sont pas impactés. Stratégie : vérification combinée origine + nonce WP valide + authentification.
 Les appels internes (Gutenberg, plugins WordPress) doivent continuer à fonctionner sans interruption.

## Logique de filtrage (Priorité)
1. Si `is_user_logged_in()`, accès autorisé.
2. Si Header `X-WP-Nonce` valide, accès autorisé (Gutenberg/Core).
3. Si le namespace est whitelisted (ex: `wc/v3`, `contact-form-7/v1`), accès autorisé.
4. Sinon, retourner `WP_Error` avec code 401 (Unauthorized).

## Whitelist Automatique
Le plugin doit proposer des presets pour :
- WooCommerce
- Contact Form 7
- Yoast SEO

## Classe : `LBS_RestApiProtection`
Fichier : `plugin/includes/Features/RestApiProtection.php`

## Stratégie de filtrage

Ne jamais bloquer sur critère IP seul (proxies, CDN). Combiner :

1. **Nonce WP valide** (`wp_verify_nonce`) → autoriser
2. **Utilisateur authentifié** (`is_user_logged_in()`) → autoriser
3. **Endpoint en whitelist** → autoriser
4. **IP en whitelist** (optionnel) → autoriser
5. Tout le reste → HTTP 401

## Hook WordPress

```php
add_filter('rest_authentication_errors', function($result) {
    // Logique de filtrage ici
    return $result; // ou new WP_Error('rest_forbidden', ..., ['status' => 401]);
});
```

## Config JSON

```json
"rest_api": {
  "enabled": true,
  "whitelist_endpoints": ["/wp-json/wp/v2/posts"],
  "whitelist_ips": []
}
```

## Endpoints WordPress natifs à ne jamais bloquer (internes)

- Tout appel avec nonce `wp_rest` valide (Gutenberg)
- `/?rest_route=` (compatibilité)
- Cron WordPress interne

## Contraintes

- Gutenberg envoie ses requêtes avec un nonce `wp_rest` dans le header `X-WP-Nonce` → toujours laisser passer
- WooCommerce utilise l'API REST en interne → tester la compatibilité
- Ne pas renvoyer 403 mais 401 pour ne pas révéler que la route existe et est protégée

## Critères d'acceptation

- [ ] Requêtes externes non authentifiées → HTTP 401
- [ ] Requêtes internes avec nonce valide → HTTP 200
- [ ] WooCommerce et plugins utilisant l'API REST en interne ne sont pas bloqués
- [ ] Whitelist d'endpoints publics configurable
- [ ] Option pour autoriser des IPs spécifiques
- [ ] Un utilisateur non connecté ne peut pas lister les utilisateurs via `/wp-json/wp/v2/users`.
- [ ] L'éditeur de bloc (Gutenberg) fonctionne sans erreur de permission.
